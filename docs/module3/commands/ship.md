# Ship Command

Ship your changes by creating a branch, committing, and pushing to origin. Claude will automatically analyze the current changes and generate an appropriate branch name and description.

**Usage:** `/ship [linear-issue-id]`

**Examples:**

- `/ship` (Claude analyzes changes and ships them)
- `/ship "UIP-116"` (Claude analyzes changes and includes Linear issue in branch name)

## Implementation

1. **Analyze current changes:**

   - Run `!git status` and `!git diff` to see what files have changed
   - Analyze the changes to understand what feature/fix is being implemented
   - Group changes by logical units (e.g., separate commits for different components, features, or types of changes)
   - Generate a concise, descriptive branch name based on the overall changes

2. **Get user information:**

   - Try to get git username with `!git config user.name` or fallback to `!whoami`
   - Convert username to lowercase for branch naming (ensure all branch names use lowercase)

3. **Create branch name automatically:**

   - Analyze git diff and changed files to determine the nature of changes
   - Generate descriptive branch name based on:
     - Type of change (feat/, fix/, refactor/, etc.)
     - Key components/files being modified
     - Purpose of the changes
   - With Linear issue: `{lowercase-username}/{issue-id}/{auto-generated-description}`
   - Without issue: `{lowercase-username}/{auto-generated-description}`
   - Example: `josh/UIP-116/add-user-impersonation-tracking` (auto-generated from analyzing PostHog/Sentry changes)

4. **Plan commits:**

   - Analyze if changes should be split into multiple logical commits
   - Group related changes together (e.g., all changes to a specific component, feature additions vs bug fixes, etc.)
   - Create descriptive commit messages for each logical group
   - Use appropriate prefix (feat:, fix:, refactor:, docs:, style:, test:, etc.)
   - Each commit message should be entirely generated by Claude based on the content of the changes in that group
   - Do not include Claude attribution or co-authorship in commit messages

5. **Git workflow:**

   - Check current git status with `!git status`
   - Create and checkout new branch: `!git checkout -b {auto-generated-branch-name}`
   - If there are changes to commit:

     **For single logical commit:**
     - Add all changes: `!git add .`
     - Create commit: `!git commit -m "{type}: {auto-generated description}"`

     **For multiple logical commits:**
     - For each logical group of changes:
       - Add specific files: `!git add {files-for-this-commit}`
       - Create commit: `!git commit -m "{type}: {auto-generated description for this group}"`
     - Examples of logical groupings:
       - Frontend components vs backend API changes
       - New features vs bug fixes vs documentation
       - Different modules or services
       - Setup/configuration vs implementation

   - Push branch: `!git push -u origin {auto-generated-branch-name}`

6. **Output:**
   - Show the auto-generated branch name and reasoning
   - Display commit message(s) created
   - Explain the logical grouping decision (single vs multiple commits)
   - Confirm successful push
   - Suggest next steps (create PR, continue development)

Arguments: $ARGUMENTS
